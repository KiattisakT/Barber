# PRD: Dream Catcher Booking MVP

## Overview

Dream Catcher Barber and Tattoo เป็นร้านแรกสำหรับ web app จองคิวร้านตัดผม โดยเริ่มจากร้านเดียวก่อน แต่ต้องวางโครงสร้างให้ต่อยอดเป็น SaaS หลายร้านได้ในอนาคต

ระบบต้องรองรับ 2 workflow ตั้งแต่ version แรก:

1. ลูกค้าจองคิวตัดผมได้จริงทันที
2. ลูกค้าส่งคำขอสักลายให้ร้านประเมินก่อนนัดจริง

## Goals

- ลดงานตอบแชท/รับโทรศัพท์สำหรับคิวตัดผม
- ให้ลูกค้าดูเวลาว่างและจองคิวเองได้
- ให้เจ้าของร้านเห็นคิววันนี้แบบเร็วและชัดเจน
- รองรับ walk-in หน้าร้าน
- แยก flow งานตัดผมกับงานสักลาย เพราะมีวิธีประเมินเวลาและราคาต่างกัน
- วาง `shopId` และ `staffId` ตั้งแต่แรกเพื่อรองรับหลายร้านและหลายช่างภายหลัง

## Non-Goals For MVP

- ยังไม่ทำระบบมัดจำหรือจ่ายเงินออนไลน์
- ยังไม่เชื่อม LINE Official Account จริง
- ยังไม่ทำ marketplace ค้นหาร้าน
- ยังไม่ทำ subscription สำหรับ SaaS
- ยังไม่ทำ POS เต็มรูปแบบ
- ยังไม่ทำ loyalty, membership, package หรือ commission
- ยังไม่ทำ mobile app native

## Target Users

### Customer

ลูกค้าที่ต้องการจองคิวตัดผมหรือสอบถามงานสักลาย โดยใช้งานผ่านมือถือเป็นหลัก

### Shop Owner / Staff

เจ้าของร้านหรือช่างที่ต้องจัดการคิว online booking, walk-in, และ tattoo request ในแต่ละวัน

## Initial Shop Context

ข้อมูลสาธารณะเบื้องต้นของร้าน:

- ชื่อร้าน: Dream Catcher Barber and Tattoo
- พื้นที่: สันกำแพง, เชียงใหม่
- บริการ: ตัดผมและสักลาย
- เวลาที่พบจากข้อมูลสาธารณะ: 10:00-20:00
- วันหยุดที่พบจากข้อมูลสาธารณะ: วันพุธ

ข้อมูลเวลาทำการและบริการต้องให้เจ้าของร้านยืนยันอีกครั้งก่อนใช้จริง

## MVP Scope

### Customer Booking Page

ลูกค้าเข้าหน้าจองของร้านผ่าน URL เช่น `/book/dream-catcher`

ลูกค้าสามารถ:

- เลือกหมวดบริการ `ตัดผม` หรือ `สักลาย`
- ดูข้อมูลร้านเบื้องต้น
- กรอกข้อมูลติดต่อ
- ส่ง booking หรือ request ได้จากมือถือ

### Barber Booking Flow

ลูกค้าสามารถ:

- เลือกบริการตัดผม
- ดูราคาและระยะเวลา
- เลือกวันและเวลาว่าง
- กรอกชื่อ เบอร์โทร และหมายเหตุ
- ยืนยันการจองฟรี
- ได้ booking code หรือลิงก์สำหรับอ้างอิง

ระบบต้อง:

- ป้องกันการจองซ้อนใน staff คนเดียวกัน
- เคารพเวลาทำการ วันหยุด และ blocked time
- รวม online booking กับ walk-in ใน daily queue ของร้าน

### Tattoo Request Flow

ลูกค้าสามารถ:

- ส่งคำขอสักลาย
- แนบรูป reference
- ระบุตำแหน่งบนร่างกาย
- ระบุขนาดโดยประมาณ
- ระบุ budget โดยประมาณ ถ้ามี
- เขียนรายละเอียดหรือไอเดีย
- ส่งข้อมูลให้ร้าน review ก่อนยืนยันนัด

Tattoo request เริ่มต้นด้วยสถานะ `pending_review` และยังไม่ถือว่าเป็น confirmed appointment จนกว่าร้านจะติดต่อกลับและยืนยันเวลา

### Admin Daily Queue

หน้าแรกของ admin ควรเป็น "คิววันนี้" ไม่ใช่ monthly calendar

ร้านสามารถ:

- ดูคิววันนี้ทั้งหมด
- แยกคิว online, walk-in, และ tattoo request
- เพิ่ม walk-in
- เปลี่ยนสถานะคิว
- ยกเลิกหรือแก้ไขคิว
- block เวลาไม่รับคิว

### Walk-In Support

ร้านสามารถเพิ่มลูกค้าหน้าร้านได้ทันที

ข้อมูลขั้นต่ำ:

- ประเภทบริการ
- ชื่อลูกค้า หรือใช้ค่า default เช่น "ลูกค้าหน้าร้าน"
- เบอร์โทร ถ้ามี
- บริการ
- ช่าง
- เวลาเริ่มโดยประมาณ
- หมายเหตุ

### Service Settings

ร้านสามารถจัดการบริการ:

- ชื่อบริการ
- หมวดบริการ: `barber` หรือ `tattoo`
- ราคา
- ระยะเวลา
- สถานะเปิด/ปิด
- คำอธิบายสั้น

### Business Hours And Blocked Time

ร้านสามารถตั้งค่า:

- วันเปิดร้าน
- เวลาเปิด-ปิด
- วันหยุดประจำ
- วันหยุดพิเศษ
- เวลาที่ block ไม่รับคิว
- buffer ระหว่างคิว ถ้าต้องการ

## Booking Statuses

- `pending_review`: รอร้าน review ใช้กับ tattoo request
- `confirmed`: นัดยืนยันแล้ว
- `checked_in`: ลูกค้ามาถึงร้านแล้ว
- `in_progress`: กำลังให้บริการ
- `completed`: เสร็จแล้ว
- `cancelled`: ยกเลิก
- `no_show`: ลูกค้าไม่มาตามนัด

## Product Principles

- Mobile-first เพราะลูกค้าน่าจะเข้าจากมือถือ
- Daily queue-first เพราะร้านมี walk-in
- Tattoo flow ต้องเป็น request/review ไม่ใช่ instant booking
- ข้อมูลร้านแรกต้อง harden จากการใช้งานจริงก่อนขยายเป็น SaaS
- ทุก entity หลักควรมี `shopId` แม้ตอนนี้มีร้านเดียว

## Success Metrics

- ลูกค้าสามารถจองคิวตัดผมเองได้โดยไม่ต้องทักแชท
- เจ้าของร้านเห็นคิววันนี้และเพิ่ม walk-in ได้ภายในไม่กี่คลิก
- ไม่มีการจองซ้อนในช่วงเวลาเดียวกันสำหรับช่างคนเดียว
- Tattoo request มีข้อมูลพอให้ร้านประเมินต่อได้
- ระบบพร้อมเพิ่ม staff คนที่สองโดยไม่ต้องเปลี่ยนโครงสร้างหลัก
